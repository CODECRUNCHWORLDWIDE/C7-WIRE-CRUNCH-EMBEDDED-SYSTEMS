# Week 12 — Debugging Like a Senior on the RP2040

> *The difference between a junior and a senior firmware engineer is not measured in lines of code written; it is measured in minutes-to-root-cause on a bug nobody understands. A junior re-reads the source for the fourth time and adds another `printf`. A senior attaches a probe, halts the core at the exact instruction, reads the stacked exception frame, walks the call chain back to the line that scribbled over a pointer, and has a one-paragraph runbook written before lunch. The tools that make this possible — SWD, GDB, OpenOCD, RTT, the DWT cycle counter, a logic analyzer with protocol decoders, a fault handler that persists a core image across reset — are all free or open, all run against a $4 Pico, and all reward an afternoon of deliberate practice. This week you stop printf-debugging and start engineering your way to root cause.*

Welcome to Week 12 of C7, the last week of the "Bare-Metal & RTOS" block and the week the midterm bench exam sits at the end of. For eleven weeks you have written firmware: bare-metal bring-up, GPIO and UART, linker scripts, SPI/I²C drivers, modern C++ on the metal, FreeRTOS, DMA and ring buffers, USB device classes, a signed dual-bank bootloader, and a WiFi/MQTT/TLS stack. Every one of those weeks ended with something that did not work the first time, and every one of those failures you debugged with whatever was at hand. This week we make the debugging itself the subject. You will leave with a personal runbook for the five families of firmware bug — timing, memory, peripheral-state, supply/EMI, and heisenbugs — and the muscle memory to reach for the right tool for each.

The hardware story this week is the SWD port. Every RP2040 exposes a two-wire Serial Wire Debug interface (datasheet §2.3.4, "SWD", pp. 14–15) on the `SWCLK` and `SWDIO` pads, brought out to a 3-pin debug header on the Pico board. SWD is the ARM-standard low-pin-count debug transport (ARM Debug Interface Architecture Specification ADIv5.2, IHI0031, free from ARM); over those two wires a debug probe can halt the cores, read and write every register and every byte of memory, set hardware breakpoints and watchpoints, single-step, and — critically — read SRAM *while the target runs*, which is the mechanism RTT exploits. We will drive the SWD port two ways: with a second Pico flashed as a `debugprobe` (Raspberry Pi's official CMSIS-DAP firmware, free, the standard cheap probe for this part), and conceptually with a SEGGER J-Link (the commercial gold standard, mentioned for comparison; not required). The host side is OpenOCD (open-source, BSD/GPL) talking ADIv5 to the probe and exposing a GDB remote server on `localhost:3333`, with `gdb-multiarch` (or `arm-none-eabi-gdb`) on the other end.

The Cortex-M0+ in the RP2040 is an ARMv6-M core (datasheet §2.4, pp. 16–18), and its fault model is the simplest in the Cortex-M family: there is exactly one fault exception, `HardFault` (exception number 3). Bigger cores (M3/M4/M7, ARMv7-M) split faults into `MemManage`, `BusFault`, and `UsageFault` with a rich set of status registers (`CFSR`, `MMFAR`, `BFAR`) that tell you precisely what went wrong; the M0+ has none of that. Every illegal load, every unaligned access, every jump to a non-thumb address, every divide-by-zero-that-isn't (the M0+ has no hardware divide), every stack overflow that lands in a peripheral region — all of it escalates to one `HardFault`. That sounds like a handicap and it is, but it also means the fault-handling code you write this week is short and universal: one handler, one stacked-frame layout, one unwinding strategy. We lean into it. You will write a `HardFault_Handler` in C-from-naked-assembly that captures `R0–R12`, the stacked `PC`/`LR`/`xPSR`, and a heuristic backtrace, packs them into a `cc_crash_dump_t`, persists that across the reset, and prints a human-readable postmortem on the next boot — the embedded equivalent of a Unix core dump plus `gdb -c core`.

Between the probe and the fault handler sit the two tracing technologies you will actually live in day to day. The first is **RTT** (Real-Time Transfer), SEGGER's scheme for moving `printf`-grade text off the chip with almost zero target overhead and no dedicated UART: the target writes bytes into a ring buffer in SRAM, the probe reads that ring over SWD while the core runs, and your terminal shows the log in real time. RTT survives situations a UART cannot — it works inside an ISR, it works when the system clock is wrong, it works when the only pins you have left are `SWCLK`/`SWDIO`. The genuine SWO/ITM trace path (single-wire output of the M3/M4's instrumentation trace) does **not** exist on the M0+ — the M0+ has no ITM — so on the RP2040, RTT *is* the printf-free tracing story, and that is a real and important platform fact you must not get wrong on the exam. The second tracing tool is the **DWT cycle counter** (`CYCCNT`): a free-running 32-bit counter that ticks once per CPU clock, the cheapest precise stopwatch on the chip, the tool you reach for when the bug is "this ISR sometimes takes too long." You will wrap it, calibrate out its own overhead, and use it to catch a jitter bug that no `printf` could ever see because the act of printing perturbs the very timing you are measuring.

The last third of the week leaves the chip's own introspection and goes to the bench instruments: the **logic analyzer** (a $12 8-channel USB clone running `sigrok`/PulseView, or a Saleae Logic if your lab has one) with **protocol decoders** that turn a forest of edges on SPI/I²C/UART lines into `START 0x48 W ACK 0xF4 ACK STOP`. When the bug is "the sensor returns zeros" the question is always *did my bytes leave the chip correctly, and did the peripheral answer?* — and the only instrument that answers it without trusting any of your own code is the analyzer on the wire. We do a full hung-I²C case study (the canonical "a reset mid-transaction left a slave holding SDA low forever" bug, straight from the syllabus's Week 12 lecture) and recover the bus with the nine-clock-pulse trick. Finally we put it together as a methodology — the **heisenbug**: the bug that vanishes when you observe it, because your observation (a breakpoint, a `printf`, an extra `volatile`) changed the timing or the memory layout. You will learn to reproduce heisenbugs deterministically (freeze the clocks, force the schedule, use the non-perturbing tools — RTT and DWT and the analyzer — instead of the perturbing ones), which is the single most valuable debugging skill a senior has and the one juniors most conspicuously lack.

## Learning objectives

By the end of this week, you will be able to:

- **Attach** a debug probe over SWD and drive a live GDB session. You will flash a second Pico as the Raspberry Pi `debugprobe` (CMSIS-DAP), wire its `GP2`/`GP3` to the target's `SWCLK`/`SWDIO` and a common ground, and run `openocd -f interface/cmsis-dap.cfg -f target/rp2040.cfg` to expose a GDB server on port 3333 (Raspberry Pi "Getting started with Raspberry Pi Pico", Appendix A, free PDF). From `gdb-multiarch`, you will `target extended-remote localhost:3333`, `load` an ELF, `monitor reset halt`, set hardware breakpoints (`hbreak`, because flash breakpoints need the M0+'s 4 hardware comparators — software breakpoints would require writing `BKPT` into XIP flash, which you cannot do without an erase), single-step, read memory, and dump registers. SWD vs JTAG: SWD is two pins and standard on the RP2040; JTAG is the older five-pin (`TCK`/`TMS`/`TDI`/`TDO`/`TRST`) transport, still common on FPGAs and older MCUs, and you must be able to read a JTAG pinout and explain why SWD won the Cortex-M world (ARM ADIv5.2 §B, IHI0031).
- **Script** GDB to automate the boring parts of a debug session. A `.gdbinit` with `define`d commands, Python-extension scripts (`gdb` ships an embedded CPython), `commands` attached to breakpoints, conditional breakpoints (`break foo if x > 100`), and `tbreak`/`watch`/`rwatch` for hardware watchpoints that halt the core when a specific address is written. You will write a GDB script that, on hitting a fault, automatically dumps the stacked exception frame, decodes the faulting PC to a source line via `info line *$pc`, and prints a backtrace — a one-keystroke postmortem.
- **Explain** the Cortex-M0+ exception and fault model. The eight-word stacked frame (`R0`, `R1`, `R2`, `R3`, `R12`, `LR`, return-address, `xPSR`) the core pushes on exception entry (ARMv6-M ARM §B1.5.6); the `EXC_RETURN` value in `LR` on handler entry and how its bit 2 tells you whether the faulting code was on `MSP` or `PSP` (§B1.5.8); why the M0+ collapses all fault classes into `HardFault` (no `CFSR`); how to recover the faulting instruction's address from the stacked `PC`; and the difference between a *precise* fault (the stacked PC is the offending instruction) and an *imprecise* one (write buffering means the PC has moved on — rare but real on the M0+'s store path).
- **Implement** a fault handler that captures a core dump. A `__attribute__((naked))` `HardFault_Handler` that reads the active stack pointer, passes it to a C function, captures `R4–R11` (which are *not* in the stacked frame — the handler must save them itself), records the exception number from `IPSR`, runs a heuristic stack scan to recover a backtrace (scan the stack for words that look like return addresses into the `.text` range with the thumb bit set), CRC-protects the whole record, and stores it where it survives a reset — a reserved SRAM region that the SDK's `.bss` clearing skips, or the watchdog scratch registers. On the next boot, before anything else, the code reads the record back and prints a postmortem.
- **Use** RTT for `printf`-free, non-perturbing logging. The SEGGER RTT control block (a known-magic struct in SRAM the probe scans for), a single up-channel ring buffer, the target-side `cc_rtt_write` that copies bytes into the ring and bumps the write index, and the host-side viewer (`openocd`'s `rtt server`, or `JLinkRTTViewer`) that drains the ring over SWD without halting the core. You will measure RTT's cost (a handful of cycles per byte, no I/O wait) against a blocking UART `printf` (hundreds of microseconds per line at 115200 baud) with the DWT counter, and understand *why* RTT does not perturb timing the way UART does.
- **Use** the DWT cycle counter for instruction-accurate timing. Enable `TRCENA` in `DEMCR`, enable `CYCCNTENA` in `DWT_CTRL`, read `DWT_CYCCNT` before and after a region, subtract the measurement overhead (the cost of the two reads plus the subtract, which you calibrate once), and convert cycles to nanoseconds at the known clock. You will catch a timing-jitter bug — an ISR whose worst-case latency occasionally blows its deadline — that is invisible to `printf` because printing it changes it. The M0+ DWT is reduced (no PC sampling, fewer comparators) but `CYCCNT` is present and is the load-bearing feature.
- **Capture and decode** a bus transaction on a logic analyzer. Wire a cheap 8-channel analyzer (or a Saleae) to the SPI or I²C lines, sample at ≥4× the bus clock, and apply `sigrok`/PulseView's protocol decoder (or Saleae's "high-level analyzer") to turn edges into decoded transactions. You will read an I²C capture and answer the only questions that matter: did the START/STOP framing happen, did the slave ACK its address, did the data bytes match what your driver intended, and — for the hung-bus case study — is a slave holding SDA low across a STOP that never came.
- **Diagnose and recover** a hung I²C bus. The classic failure: the master reset (or a watchdog) mid-transaction, leaving a slave that had clocked out a few bits of a read still driving SDA low, waiting for clocks that will never come; the bus is stuck because the master cannot generate a START while SDA is held low. The recovery is the nine-clock-pulse manual bit-bang (toggle SCL up to 9 times with SDA released, until the slave finishes its byte and releases SDA, then issue a manual STOP) — documented in the I²C-bus specification (NXP UM10204 §3.1.16, "Bus clear", free) and in the RP2040's I²C errata discussions. You will implement the recovery in C against the RP2040's I²C peripheral.
- **Reproduce** a heisenbug deterministically. A heisenbug changes or disappears under observation. The standard culprits: a `printf` or a breakpoint perturbs timing and hides a race; an extra `volatile` or a debug build's lack of optimization changes the memory layout and hides an uninitialized-read; an interrupt that fires at a window the debugger's halt-and-step never reproduces. The senior's toolkit: switch to non-perturbing observation (RTT, DWT, the analyzer — none of which halt the core or add I/O latency); pin the schedule (disable other interrupts, force the FreeRTOS task order, run single-core); freeze the variables (fix the clock, fix the RNG seed, fix the input); and bisect (`git bisect` plus a deterministic reproducer). You will take a supplied flaky firmware and make its bug fire on demand.
- **Perform** a postmortem from a captured core image. Given a `cc_crash_dump_t` persisted across reset (or a full SRAM image dumped over SWD with `dump binary memory core.bin 0x20000000 0x20042000`), reconstruct what the firmware was doing at the moment of the fault: identify the faulting instruction from the stacked PC, the call chain from the backtrace, the corrupted value from the stacked registers, and — when you have the matching ELF — the source line via `arm-none-eabi-addr2line -e firmware.elf 0x10001234`. This is exactly the Unix `gdb program core` workflow, transplanted to a chip with 264 KB of SRAM and no operating system.
- **Author** a debugging runbook. The midterm and the real world both reward a written procedure: "symptom X → first reach for tool Y → if that shows Z, the cause is almost always W." You will distill the week's five bug families into a one-page runbook with a decision tree (does the core fault? → core dump; does it hang silently? → halt over SWD and read `$pc`; does the bug move when observed? → non-perturbing tools; is the data wrong on the wire? → analyzer; is the timing wrong? → DWT) and a tool-per-symptom table. This runbook is a graded deliverable and the thing you will actually carry into your career.

You have skimmed the **ARMv6-M Architecture Reference Manual §B1.5 ("Exceptions", in DDI0419)**, free from ARM at <https://developer.arm.com/documentation/ddi0419/latest/>. The two pages you will refer to constantly are the stacked-frame layout (§B1.5.6) and the `EXC_RETURN` encoding (§B1.5.8). Have it open in a tab.

You have OpenOCD (build the `rp2040` branch from <https://github.com/raspberrypi/openocd> or install a distro package new enough to know the RP2040 — `brew install openocd` on macOS gives you a recent one), `gdb-multiarch` (Debian/Ubuntu `apt install gdb-multiarch`; macOS `brew install arm-none-eabi-gdb`), `picotool`, `sigrok-cli`/`pulseview` (<https://sigrok.org/>), and your Pico SDK build environment from the previous eleven weeks. A `requirements.txt` in `mini-project/` pins the host-side Python.

You have a **logic analyzer**. A $12 8-channel "Saleae clone" (CY7C68013A / FX2-based) running under `sigrok` is sufficient for this week's bus speeds (I²C at 100–400 kHz, SPI at a few MHz). A genuine Saleae Logic 8 is nicer (higher sample rate, the polished "high-level analyzer" plugins) but not required. An oscilloscope is mentioned for analog-domain bugs (supply droop, EMI, slow edges) but is optional — most of this week's bugs are digital.

## Topics covered

- **SWD and the ADIv5 debug architecture.** Two wires (`SWCLK`, `SWDIO`), a packet protocol carrying Debug Port (DP) and Access Port (AP) reads/writes, the MEM-AP that gives the probe a window into the entire memory map, and the Debug Core Register Selector/Data registers (`DCRSR`/`DCRDR`) that read and write the CPU's `R0–R15`/`xPSR` while the core is halted. Cite RP2040 datasheet §2.3.4 and ARM ADIv5.2 (IHI0031). SWD vs JTAG: SWD's two pins vs JTAG's five (`TCK`/`TMS`/`TDI`/`TDO`/optional `TRST`); JTAG's daisy-chain (multiple devices on one scan chain) vs SWD's single-target focus; why every modern Cortex-M brings out SWD and treats JTAG as legacy.
- **OpenOCD and the GDB remote protocol.** OpenOCD speaks ADIv5 to a CMSIS-DAP/J-Link probe on one side and the GDB Remote Serial Protocol (`$g`, `$m`, `$M`, `$c`, `$s`, `$Z`/`$z` packets) on a TCP port on the other. `monitor` commands in GDB are passthroughs to OpenOCD (`monitor reset halt`, `monitor rtt setup`, `monitor flash write_image`). The RP2040 target config (`target/rp2040.cfg`) handles the dual-core declaration — the RP2040 exposes two M0+ cores on the SWD multidrop, and OpenOCD configures both `rp2040.core0` and `rp2040.core1`.
- **GDB scripting.** `.gdbinit`, `define`/`document`, breakpoint `commands`, conditional and temporary breakpoints, hardware watchpoints (`watch`/`rwatch`/`awatch`) backed by the M0+'s 2 DWT comparators, the `tui` text UI, `x` examine, `info registers`, `p/x`, and the embedded Python API (`gdb.execute`, `gdb.parse_and_eval`, custom commands as `gdb.Command` subclasses). The killer app: a Python GDB command that, on a fault, reads the stacked frame and prints a decoded postmortem in one keystroke.
- **The Cortex-M0+ fault model.** `HardFault` (exc 3) as the universal fault. The stacked 8-word frame. `R4–R11` are callee-saved and *not* stacked by hardware — the fault handler must capture them itself if it wants a full register set. `IPSR` (low 9 bits of `xPSR`) gives the active exception number. `EXC_RETURN` in `LR` on entry: `0xFFFFFFF9` (MSP, thread), `0xFFFFFFFD` (PSP, thread), `0xFFFFFFF1` (MSP, handler). Precise vs imprecise faults and why the M0+'s simple store path makes almost all of its faults precise. No `MemManage`/`BusFault`/`UsageFault`, no `CFSR`/`MMFAR`/`BFAR` — this is the platform fact the exam tests.
- **Persisting a crash across reset.** Three mechanisms: (1) a reserved no-init SRAM region (`__attribute__((section(".noinit")))` plus a linker `NOLOAD` section the C runtime's `.bss` clear skips) — survives a soft reset but not a power cycle; (2) the watchdog scratch registers (`WATCHDOG_SCRATCH0..7`, eight 32-bit words that survive any reset short of power-off) — enough for a magic + a faulting PC + an exception number, the cheapest dump; (3) a flash page — survives power-off but costs an erase in the fault handler, which is risky (the chip may be in a state where flash writes are unsafe). We use no-init SRAM for the full dump and scratch registers for the minimal one.
- **RTT (Real-Time Transfer).** The control block with magic `"SEGGER RTT"`, the up-channel ring buffer, the target writing bytes and bumping `write_offset`, the host probe reading SRAM over SWD and bumping `read_offset`, the wrap-around arithmetic, and the "skip vs block when full" policy. Why RTT works in an ISR (no syscalls, no blocking, no shared peripheral lock) and why it does not perturb timing (no busy-wait on a UART FIFO). The RP2040 has no ITM/SWO, so RTT is the printf-free path; SWO is an M3/M4+ feature and a common wrong-answer on the exam.
- **The DWT cycle counter.** `DEMCR.TRCENA`, `DWT_CTRL.CYCCNTENA`, `DWT_CYCCNT`. The free-running 32-bit counter at the core clock (125 MHz default → 8 ns resolution, ~34.4 s wrap). Measuring a region (read, region, read, subtract overhead). Converting cycles to time. The M0+'s reduced DWT: `CYCCNT` present, PC sampling and the full comparator bank absent. Using it to find a jitter bug that `printf` hides.
- **Logic-analyzer capture and protocol decoding.** Sample rate vs bus rate (Nyquist plus margin: ≥4× the fastest edge of interest), channel-to-signal mapping, trigger setup (rising-edge on CS, or a pattern), and the `sigrok` decoder stack (`i2c`, `spi`, `uart`, then a stacked decoder like `i2c` → a sensor-specific one). Reading the decoded transcript and correlating it against your driver's intended sequence. The analyzer is the instrument that does not trust your code.
- **The hung-I²C case study.** Why a mid-transaction reset hangs the bus (slave still driving SDA low, waiting for clocks); why the master cannot recover with a normal START (START requires SDA to fall while SCL is high, impossible if SDA is already low); the nine-clock recovery (NXP UM10204 §3.1.16); and the C implementation against the RP2040 I²C peripheral (`i2c0`/`i2c1`, the DesignWare IP — datasheet §4.3, pp. 446–490). We also cover the analyzer capture that *proves* the slave is the one holding SDA.
- **Heisenbugs.** The observer effect in firmware. Timing-perturbing observations (breakpoints, `printf`, halts) vs non-perturbing ones (RTT, DWT, the analyzer). Layout-perturbing changes (`volatile`, optimization level, stack canaries) that hide uninitialized-memory and stack-overflow bugs. Strategies: pin the schedule, freeze the variables, use non-perturbing tools, bisect with a deterministic reproducer. The single most senior skill on the syllabus.
- **The runbook discipline.** A decision tree from symptom to tool to likely cause, a tool-per-symptom table, and the habit of writing the runbook *during* the debug while the state is fresh. This is the artifact that scales a senior across a team.

## What is intentionally out of scope this week

- **ETM/ETB full instruction trace.** Real instruction trace (every executed instruction streamed off-chip) needs an Embedded Trace Macrocell and a trace port, which the Cortex-M0+ does not have and the RP2040 does not expose. On M7 parts with ETM plus a trace probe (a J-Trace, a Lauterbach) you get cycle-accurate execution history; on the M0+ you get hardware breakpoints, watchpoints, `CYCCNT`, and RTT, and that is what we teach. Do not expect SWO/ITM or ETM on this chip — getting that platform fact right is worth a quiz point.
- **Commercial trace tools (Lauterbach TRACE32, SEGGER Ozone deep features).** Excellent, expensive, and out of reach for a $4-board course. We mention Ozone and SystemView (SEGGER's free-for-non-commercial tools, which *do* run against a J-Link on the RP2040) as the upgrade path, but the graded work is all on free/open tooling (OpenOCD, GDB, sigrok).
- **FPGA/SoC JTAG boundary scan.** JTAG's original purpose (boundary-scan board test, IEEE 1149.1) and its use on FPGAs is adjacent and important but not a Cortex-M debugging topic. We cover JTAG only enough to contrast it with SWD and read a pinout.
- **Formal timing analysis / WCET tooling.** We measure worst-case execution time empirically with the DWT counter and a stress harness; we do not do static WCET analysis (aiT, Bound-T). For hard-real-time certification that matters; for finding and fixing a jitter bug, the empirical approach is what a working engineer reaches for first.
- **Power and EMI debugging in depth.** We name the supply-droop and EMI bug family and point at the oscilloscope and the near-field probe, but the analog-domain forensics (decoupling, ground bounce, crosstalk) is a hardware course. This week's bugs are overwhelmingly digital — the right altitude for a firmware track.

## A note on the printf reflex

Every engineer reading this debugs with `printf`. It is not shameful; it is the universal first tool, it requires nothing but a UART, and for a great many bugs it is exactly right. The point of this week is not to take `printf` away — it is to make you *notice* when `printf` is the wrong tool and to have the next three tools ready. `printf` is wrong when the bug is in the timing (printing changes the timing), when the bug is in an ISR (printing from an ISR is a deadlock or a re-entrancy bug waiting to happen), when the system has already faulted (there is no `printf` after a HardFault unless you built one into the handler), and when the bug only appears in the optimized release build (adding a `printf` changes the optimization and the bug moves). For each of those, this week gives you the senior's answer: RTT for ISR-safe non-perturbing logging, DWT for timing the thing you cannot print, the fault handler for the postmortem after the crash, and the analyzer for the truth on the wire. Keep `printf`. Add the rest.

The midterm is at the end of this week, and it is a bench exam, not a written one. The reviewer hands you an unfamiliar board and a datasheet and watches you work. Nobody passes that exam by re-reading source — you pass it by attaching the probe, reading the registers, and reasoning from what the chip actually tells you. That is the skill this whole week exists to build.
