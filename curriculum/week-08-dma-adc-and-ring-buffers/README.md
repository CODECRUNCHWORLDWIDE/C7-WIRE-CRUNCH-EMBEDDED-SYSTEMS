# Week 8 — DMA, ADC Streaming, and Ring Buffers on the RP2040

> *The CPU is a poor data-mover. It is a thinking organ; it should not be made to copy bytes from a peripheral to memory one word at a time when there is a piece of silicon nearby whose entire purpose is to copy bytes from a peripheral to memory one word at a time. The DMA controller is that silicon. Once you have wired it up correctly, the CPU stops touching the data path entirely — it touches the data, in chunks, at a rate of its own choosing.*

Welcome to Week 8 of C7. Last week you replaced a polling loop with an EXTI line and a deferred-interrupt task. The MPU-6050 ran at 200 Hz, the ISR was three lines, the stream buffer carried 14 bytes per frame, and your Saleae trace put the worst-case interrupt-to-task latency at 4.2 µs. The CPU spent the rest of its time in `__WFI`. That architecture works fine up to a few thousand interrupts per second. It stops working at higher rates.

This week you find the rate where it stops working, and you measure why. The RP2040's ADC can run continuously at up to 500 kSample/s with one channel selected, or 100 kSample/s round-robined across multiple channels (datasheet §4.9.3, pp. 565–567). At 500 kSample/s, an ADC-FIFO-empty interrupt would fire every 2 µs. The Week 7 architecture would burn 4.2 µs of ISR-to-task work to service every one — meaning the ISR latency *exceeds the sample interval* and the FIFO fills, then overruns, then drops samples silently. Polling is no better. Both architectures lose; both lose for the same reason. The CPU cannot keep up.

The DMA controller solves it. You configure one of the RP2040's twelve DMA channels (datasheet §2.5, pp. 102–144) to read from the ADC FIFO at the rate the ADC's `DREQ` signal demands, and to write into a 4 KB region of SRAM. The CPU is not involved in any of those transfers. You give the DMA channel a second buffer's worth of work via *channel chaining*: when channel A finishes its 1024-sample block, it triggers channel B; when B finishes, B triggers A; the pair runs forever and the ADC stream lands in alternating halves of an 8 KB ring without a single CPU cycle spent on a copy. A FreeRTOS task wakes on a transfer-complete IRQ once every 1024 samples (every 2 ms at 500 kS/s), drains the just-completed half, processes it, and goes back to sleep. The ISR-to-task latency budget is now ~2 ms instead of ~2 µs — three orders of magnitude of slack.

The deeper material this week is the *DMA mental model*. Three things move: the data, the control words, and the timing pulses (DREQs). The DMA controller has a register file per channel; the ADC has a DREQ wired to a specific channel-trigger bit; the SDK has a thin C wrapper over that register file (`hardware_dma`). You will learn to read the channel control word (`CTRL`) field by field — `DATA_SIZE`, `INCR_READ`, `INCR_WRITE`, `RING_SIZE`, `RING_SEL`, `TREQ_SEL`, `CHAIN_TO`, `IRQ_QUIET` — because every bug you will hit this week is a CTRL field misset by one bit. You will learn that ADC and DMA are connected through a single hardware FIFO and a single DREQ line; misconfigure either side and the FIFO either drains too slowly (ADC overrun, samples lost) or too fast (DMA reads garbage on FIFO-empty).

The mini-project ties Week 7 and Week 8 together: a 1 MS/s ADC capture into a 4 KB ring buffer, with channel-chained DMA writing and a FreeRTOS task consuming 1024-sample chunks for an FFT-friendly power-spectrum estimate. The capture runs for 60 s with zero overruns measured. Pass criterion is a Saleae GPIO toggle on every chunk-complete IRQ and zero gaps in the toggle train under continuous load.

## Learning objectives

By the end of this week, you will be able to:

- **Explain** the RP2040 DMA controller's architecture: 12 independent channels, each with a 32-bit read pointer, a 32-bit write pointer, a 32-bit transfer count, and a 32-bit control word (`CTRL`); a shared arbiter; a per-channel IRQ line routed to one of two NVIC slots (`DMA_IRQ_0` or `DMA_IRQ_1`). Cite RP2040 datasheet §2.5.1 (overview) pp. 102–108, §2.5.2 (channel registers) pp. 108–119, §2.5.3 (channel control) pp. 119–129, and the pico-sdk `hardware_dma` reference at <https://www.raspberrypi.com/documentation/pico-sdk/group__hardware__dma.html>.
- **Decode** the `CH<n>_CTRL_TRIG` register field by field — `DATA_SIZE` (00 byte, 01 halfword, 10 word; matches FIFO width on the peripheral side), `INCR_READ` (advance the read pointer after each transfer; off for peripheral-source channels), `INCR_WRITE` (advance the write pointer; on for memory-destination channels), `RING_SIZE` and `RING_SEL` (auto-wrap the read or write pointer at a power-of-two boundary — this is the hardware ring buffer feature), `TREQ_SEL` (which DREQ line gates the transfer; 36 for ADC, 0–11 for chained channels), `CHAIN_TO` (which channel to trigger when this one's count reaches zero), `IRQ_QUIET` (suppress the IRQ on completion when the channel is meant to be invisible). Cite §2.5.7 (control register details) pp. 129–134.
- **Wire** the ADC's hardware FIFO to a DMA channel via DREQ. The ADC has an internal 8-entry, 16-bit-wide FIFO (datasheet §4.9.2.1, pp. 562–563); when the FIFO has at least one sample, it asserts `DREQ_ADC` (DREQ index 36); the DMA channel configured with `TREQ_SEL=36` issues exactly one transfer per assertion. Configure the channel with `DATA_SIZE=halfword`, `INCR_READ=0` (the FIFO is at one address), `INCR_WRITE=1` (the SRAM destination advances). Cite pico-sdk `hardware_adc` at <https://www.raspberrypi.com/documentation/pico-sdk/group__hardware__adc.html> and `adc_fifo_setup`, `adc_run`, `adc_select_input` in particular.
- **Implement** a single-channel free-running ADC capture: `adc_init()`, `adc_gpio_init(26)` for ADC0 (GP26), `adc_select_input(0)`, `adc_set_clkdiv(0)` for full 500 kS/s, `adc_fifo_setup(true, true, 1, false, false)` to enable FIFO + DREQ + threshold 1 + no error bit + no byte-shift, `dma_channel_configure(...)`, `dma_channel_start(...)`, `adc_run(true)`. Measure the realized sample rate by timestamping the IRQ that fires when a 1024-sample DMA block completes — at 500 kS/s, this should fire every 2048 µs +/- 1 µs (the SDK rounds the ADC clock divider to integer cycles of the 48 MHz ADC clock; document the rounding).
- **Pace** higher transfer rates with the system timer. The ADC alone is capped at 500 kS/s by its 48 MHz reference and the 96-cycle conversion (datasheet §4.9.3, p. 565). The mini-project's "1 MS/s" target therefore uses *two* ADC inputs interleaved with `adc_set_round_robin(0x03)` at the ADC's full rate per channel — 500 kS/s per channel, 1 MS/s aggregate. Compute the per-channel Nyquist (250 kHz) and the aliasing implication on your input filter. Cite §4.9.4 (round-robin), pp. 567–568.
- **Chain** two DMA channels for ping-pong ring-buffer operation. Channel A's `CHAIN_TO` points at channel B; B's `CHAIN_TO` points at A. Each channel's transfer count is 1024 halfwords (2 KB). The 4 KB ring is then two 2 KB halves; A writes the first half, B writes the second, and on completion of either the corresponding `DMA_IRQ_0` line fires for the *just-completed* channel — the kernel wakes a task that reads the half that is not currently being written. Both channels' write pointers must be re-armed in the IRQ handler (the count and pointer registers are consumed by the transfer). Cite §2.5.6.2 (chain-to mechanics) pp. 127–129 and §2.5.4 (interrupts) pp. 122–124.
- **Apply** the RP2040 DMA's hardware ring-buffer feature as an alternative to manual chaining. Setting `RING_SIZE=N` and `RING_SEL=1` (wrap the write pointer) makes the channel's write pointer auto-wrap modulo 2^N bytes — a single channel can then write into a 4 KB ring forever without re-arming, with the transfer count set to a large value (or to a "ring of one" by leaving count at the ring size). Trade-off: you lose the natural "half-full" / "full" IRQ boundaries that chaining gives you. Most production code uses chaining for the boundary semantics; hardware ring is sweet for simpler streams (a UART log). Cite §2.5.3.2 (ring) p. 121.
- **Reason** about the difference between an ISR-driven sample reader and a DMA-driven sample reader. At 500 kS/s the per-sample budget is 2 µs. The Cortex-M0+ at 125 MHz services its fastest ISR in ~28 cycles (~225 ns) plus the ISR body. A read of the ADC FIFO, store to SRAM, increment pointer, modulo-mask is ~12 cycles in tight assembly (~96 ns) — total ~320 ns per sample if the ISR fires every sample. That looks feasible on paper. In practice, *any* other ISR (SysTick at 1 kHz, the FreeRTOS tick) preempts and adds tens of µs of jitter; the ADC FIFO is 8 deep, so a 16 µs hiccup overruns it. DMA is not faster than the ISR on a per-sample basis — it is *isochronous*, immune to the tick. Make this comparison numerically in Lecture 1.
- **Apply** the producer/consumer ring-buffer pattern with one writer (the DMA) and one reader (a FreeRTOS task). The two indices (`head`, `tail`) need careful synchronization: the DMA does not know the task's `tail`; the task does not know the DMA's `head` except by reading the channel's `TRANS_COUNT` register, which is decremented in hardware on every transfer. The pattern is: in the chain-complete IRQ, `head = (head + 1024) & (RING_SIZE - 1)`; in the task, `tail = (head_at_wake - 1024) & (RING_SIZE - 1)`; the consumer reads `[tail, tail+1024)`. Treat this carefully because any sloppiness produces silent corruption — the DMA does not stop on overrun, it overwrites the consumer's tail.
- **Detect** overrun. The ADC peripheral sets an error bit in the top byte of each FIFO entry when its FIFO overflows (datasheet §4.9.2.4, p. 564). If `adc_fifo_setup`'s `err_in_fifo` parameter is `true`, this bit is preserved and your consumer can mask-check it (`(sample >> 15) & 1u`). For DMA-side overrun (consumer too slow), you instrument it: at chain-complete IRQ, check whether the consumer's `tail` matches the new `head` modulo half a ring — if so, the consumer is one buffer behind and the next IRQ will run it over. Increment an overrun counter and log it.
- **Measure** end-to-end latency and throughput on bench. Toggle GP14 in the chain-complete IRQ, toggle GP13 when the consumer task begins draining, toggle GP12 when it finishes. The three edges on the Saleae give you (a) the IRQ-to-task latency, (b) the per-chunk processing time, (c) the slack before the next chain-complete IRQ. A pass run shows three clean trains at the expected rates (488 Hz IRQ, ~488 Hz wake, ~488 Hz finish — 500 kS/s ÷ 1024 samples = 488.28 Hz) with no missed teeth.
- **Build** an end-to-end DMA-streaming firmware: ADC at 1 MS/s aggregate, two DMA channels in ping-pong, a 4 KB SRAM ring, a FreeRTOS consumer at priority 4 computing a running RMS and peak detector on 1024-sample chunks, a UART gatekeeper at priority 2 printing one summary line per second. Budget: 1 hour of continuous capture, 0 overruns measured, < 50 % CPU utilization. The mini-project is the deliverable; the bench artifact is the Saleae capture plus an `OVERRUN-COUNTERS.md` showing zeros.

## Topics covered

- **The DMA controller as a data-mover peripheral.** The RP2040's DMA is twelve independent channels (datasheet §2.5.1, p. 103), each capable of moving 8/16/32-bit transfers between any pair of memory or peripheral addresses, at a rate controlled by a *DREQ* (data request) signal from the source or destination peripheral. A channel's state lives in five registers: `READ_ADDR`, `WRITE_ADDR`, `TRANS_COUNT`, `CTRL`, and an aliased `CTRL_TRIG` whose write starts the channel. The arbiter between channels is round-robin by default; channel priority is configurable.
- **The DREQ mechanism.** Every DMA-capable peripheral has a DREQ line wired into the DMA controller; the ADC's DREQ is index 36 (datasheet §2.5.3.1 table, p. 121). When the peripheral has work for the DMA (e.g., the ADC has at least one entry in its FIFO), it asserts DREQ; when the DMA completes a transfer it deasserts DREQ for one cycle and the peripheral re-asserts if more work remains. This pacing means the DMA cannot read faster than the peripheral produces — no ADC underrun is possible.
- **The ADC peripheral.** RP2040's ADC is a 12-bit successive-approximation converter (datasheet §4.9, p. 558) with four external channels (GP26 ADC0, GP27 ADC1, GP28 ADC2, GP29 ADC3 — the last shared with the Pico's onboard VSYS divider) plus an internal temperature sensor (ADC4). Maximum sample rate is 500 kS/s on a single channel; round-robin across multiple channels divides the rate evenly (e.g., 250 kS/s per channel with two enabled). The FIFO is 8 entries, 16-bit wide (12 data bits + 1 error bit + 3 reserved); the DREQ trigger level is configurable 1–8.
- **DMA channel chaining.** Setting `CHAIN_TO=K` on channel J makes channel K's `CTRL_TRIG` register be written automatically when J's transfer count hits zero (datasheet §2.5.6.2, p. 127). The chain target's read/write/count registers must have been pre-loaded; the chain trigger only restarts the channel. A pair `A → B → A` forms a ping-pong with zero CPU intervention.
- **Hardware ring buffers via `RING_SIZE` / `RING_SEL`.** Setting `RING_SIZE=N` and `RING_SEL=1` makes the channel's *write* pointer wrap at address boundaries of 2^N bytes — i.e., the bottom N bits of the write pointer roll over while the upper bits stay fixed. The destination buffer must be aligned to its size. This is a hardware feature, not a software pattern; one channel + ring wrap can stream forever without any IRQ.
- **The chain-vs-ring decision.** Chaining gives you a clean per-half IRQ — the consumer wakes on every half-fill, which is the natural boundary for processing fixed-size chunks. Ring wrap gives you the absolute minimum overhead — one channel, no IRQ, no boundary — but you must measure the channel's `TRANS_COUNT` in the consumer to know how far the writer has advanced. For DSP-style 1024-sample-chunk processing, chaining wins. For a "telemetry to UART" log stream, ring wins.
- **The single-producer-single-consumer ring buffer.** The classic Lamport circular buffer (Lamport 1977, "Concurrent Reading and Writing"): one writer advances `head`, one reader advances `tail`, both wrap at the buffer size. Lamport proved that on a sequentially-consistent memory model, no locks are needed — the writer only reads `tail` and writes `head`, the reader only reads `head` and writes `tail`, and the wrap test (`head != tail` for non-empty, `((head+1) % N) != tail` for non-full) is correct under the natural read-after-write ordering. The Cortex-M0+ has no out-of-order execution and a single store buffer; the natural ordering holds. We will not need atomics this week. (Multi-core would.)
- **Overrun semantics.** The ADC FIFO sets an error bit on overflow but does not stop sampling; it overwrites the oldest entry. The DMA does not check whether the consumer has read the buffer before it writes the next half; it just writes. The pattern for detecting overrun on the DMA side is to track the consumer's expected tail relative to the producer's head in the chain-complete IRQ and to log + increment when they collide.
- **Power profile.** With DMA carrying the ADC stream and the CPU idle in `__WFI` between chunk-complete IRQs, the system spends ~95 % of wall-clock with the core halted. At 1 MS/s aggregate the consumer task wakes 976 times per second, processes 2048 samples per second, and finishes each chunk in ~120 µs. Net 3V3 current drops from ~85 mA (polled equivalent) to ~52 mA at 125 MHz — a 39 % power saving on top of Week 7's 30 %.
- **Common DMA bugs.** (1) `INCR_WRITE=0` when the destination is memory — the channel hammers the same word and you see one sample repeated. (2) `INCR_READ=1` when the source is a peripheral FIFO — the read pointer marches past the FIFO and you read register garbage. (3) `DATA_SIZE` mismatched to the ADC FIFO width (the FIFO is 16-bit; if you read 32-bit, you get one valid sample and one stale half-word per transfer). (4) Channel's IRQ not enabled in `INTE0` (the channel's DMA finished, but `dma_irqn_set_channel_enabled` was never called). (5) Forgetting to re-arm `WRITE_ADDR` in the chain-complete IRQ — the second half writes into where the first half ended, not into the buffer's other half. We will hit and fix all five.

## A note on the on-device compile

The exercises and the mini-project are written to compile under the pico-sdk + FreeRTOS-Kernel build set up in Week 6. The course does not ship a turnkey CMake recipe per file because the SDK's CMakeLists conventions differ enough across host platforms (macOS Homebrew vs Ubuntu apt vs Windows MSYS2) that a single fixed recipe would mislead. Each `.c` file has a comment block listing the pico-sdk libraries it expects to link (`pico_stdlib`, `hardware_dma`, `hardware_adc`, `FreeRTOS-Kernel-Heap4`, etc.) and the GPIO pin map it assumes. The `mini-project/README.md` shows a worked CMakeLists for the full target; copy it into a fresh project directory, drop the `.c` and `.h` files in, run `pico_sdk_init()`, and `cmake -B build && cmake --build build` should produce a `.uf2`. On-device compile and flash is a student responsibility this week as in Week 6 and Week 7.
