# Week 9 — USB Device Classes on the RP2040 with TinyUSB

> *USB is two protocols stacked on a wire. Underneath there is a chip-to-chip serial link that the silicon engineers wrote — packets, NAKs, ACKs, toggle bits, four-bit PIDs. On top there is a protocol the operating-system engineers wrote — descriptors the host reads to find out what the device is, control transfers on endpoint zero that negotiate the connection, class-specific protocols for keyboards and disks and serial ports. The wire layer you do not touch. The descriptor layer is where you do all your work. TinyUSB is the library that sits between the two and lets you write the descriptor layer in C without ever opening the USB 2.0 spec to Chapter 8.*

Welcome to Week 9 of C7. Last week the RP2040's DMA controller was your data-mover and the ADC was your sample source; a 4 KB SRAM ring buffer carried a 1 MS/s capture stream from peripheral to FreeRTOS task with zero CPU cycles spent on the copy. You learned to read a peripheral's register file, wire it to a DREQ line, and let the silicon handle the data path. This week the data path is the USB bus and the silicon is the RP2040's full-speed USB device controller — but the *register file* you will read is not in the datasheet. It is the USB 2.0 specification's Chapter 9, the chapter that describes the four kinds of descriptor that every USB device on Earth presents to its host during enumeration.

USB device work has two layers. The lower layer is *enumeration*: when you plug a Pico into your laptop, the laptop's USB host controller pulls D+/D- low, waits for the device to attach, resets it, assigns it address 1 through 127, asks it "what are you?", and the device replies with its descriptors. This conversation happens entirely on endpoint zero — the only endpoint guaranteed to exist on any USB device — using control transfers. The host's questions are standard, defined in USB 2.0 §9.4 (eight requests: GET_DESCRIPTOR, SET_ADDRESS, SET_CONFIGURATION, etc., pp. 250–263 of the spec). The device's answers are byte arrays formatted exactly per USB 2.0 §9.6 (the descriptor table, pp. 261–272). Get either side wrong by one byte and the device fails to enumerate; the host shows it as "USB device descriptor request failed" in Device Manager and you spend an evening with Wireshark's USBPcap dissector counting bytes.

The upper layer is *class behavior*: once enumerated, the device speaks a class-specific protocol on its non-zero endpoints. CDC ACM (Communications Device Class, Abstract Control Model) is a virtual serial port — your code calls `tud_cdc_write(buf, len)`, the host's driver presents `/dev/cu.usbmodem...` or `COM5`, and a terminal-emulator user sees bytes. HID (Human Interface Device) is the class that keyboards, mice, and game controllers use — your code calls `tud_hid_report(report_id, &report, sizeof report)`, the host's HID parser unpacks the report against a *report descriptor* you supplied during enumeration, and the OS routes the data to the input subsystem. MSC (Mass Storage Class) is what makes a USB stick a drive — your code answers SCSI READ_10 and WRITE_10 commands against a backing store (here, a small RAM disk), and the host mounts a FAT12 filesystem.

TinyUSB is the upstream C library that pico-sdk vendors in to give you these classes without writing them yourself. It is single-author originally (Ha Thach) and maintained by a community; the pico-sdk pulls it in as `tinyusb` and exposes a thin wrapper (`pico_stdio_usb` for the case where you only want CDC, or direct TinyUSB API for the full set). You will work the direct API this week, because the moment you want HID alongside CDC the `pico_stdio_usb` convenience layer is no longer enough.

The mini-project is a *composite* device: one Pico enumerates as a CDC virtual serial port *and* a HID keyboard *and* a 64 KB RAM-disk MSC drive at the same time. The host sees one USB device with three interfaces; the laptop's terminal app opens the CDC node; the laptop's text editor receives keystrokes from the HID interface when you press the BOOTSEL button; Finder or File Explorer mounts the RAM disk and you can drag a file onto it. The composite descriptor is the load-bearing artifact — it is ~200 bytes of byte-perfect data, and one wrong byte breaks enumeration entirely. We build it field-by-field with explicit `uint8_t` constants.

## Learning objectives

By the end of this week, you will be able to:

- **Explain** the USB 2.0 device enumeration sequence from `VBUS` rise through `SET_CONFIGURATION`. USB 2.0 §9.1.1 (Connected, Powered, Default, Address, Configured, Suspended) defines the six device states (pp. 239–243). The host's actions in order are: bus reset (SE0 for at least 10 ms), `GET_DESCRIPTOR(Device, 0, 8 bytes)` to learn `bMaxPacketSize0`, second bus reset, `SET_ADDRESS(addr)` with addr in 1..127, `GET_DESCRIPTOR(Device, addr, 18 bytes)`, `GET_DESCRIPTOR(Configuration, addr, 9 bytes)` to learn the configuration's total length, `GET_DESCRIPTOR(Configuration, addr, wTotalLength)` to get the full descriptor tree, then `GET_DESCRIPTOR(String, addr, ...)` for each non-zero string index, and finally `SET_CONFIGURATION(1)`. Cite USB 2.0 §9.1.2 (bus enumeration, pp. 243–245) and §9.4 (standard device requests, pp. 250–263).
- **Decode** a USB device descriptor byte-by-byte. The 18-byte device descriptor format is fixed (USB 2.0 §9.6.1, pp. 261–263): `bLength=18, bDescriptorType=1, bcdUSB (2 bytes, little-endian, e.g. 0x0200 for USB 2.0), bDeviceClass, bDeviceSubClass, bDeviceProtocol, bMaxPacketSize0 (8/16/32/64 for full-speed; 64 for high-speed), idVendor (2 bytes LE), idProduct (2 bytes LE), bcdDevice (2 bytes LE), iManufacturer, iProduct, iSerialNumber, bNumConfigurations`. For a composite device using Interface Association Descriptors, `bDeviceClass=0xEF, bDeviceSubClass=0x02, bDeviceProtocol=0x01` (Microsoft's "Interface Association" miscellaneous-class trick — USB-IF Engineering Change Notice 2005-09-26).
- **Decode** a configuration descriptor and its trailing interface, endpoint, and class-specific descriptors. The configuration descriptor (USB 2.0 §9.6.3, pp. 264–266) is followed by a flat byte array of interface descriptors (§9.6.5, pp. 268–269), endpoint descriptors (§9.6.6, pp. 269–271), and class-specific descriptors (CDC, HID, MSC) interleaved in a strict order. `wTotalLength` is the sum of every descriptor in the chain — get this wrong and the host stops reading mid-tree. We will build this for the composite device byte-by-byte with `#define` constants for every length and offset.
- **Configure** TinyUSB's two callback families. The *device-class callbacks* are the ones TinyUSB invokes on you: `tud_cdc_rx_cb(itf)` when the host sends bytes to your CDC OUT endpoint, `tud_hid_get_report_cb(...)` when the host asks for a HID report via the control pipe, `tud_msc_read10_cb(...)` for a SCSI read against your RAM disk. The *device-stack callbacks* are higher-level: `tud_mount_cb()` when the host issues `SET_CONFIGURATION`, `tud_umount_cb()` on disconnect, `tud_suspend_cb()` on bus suspend. Both families live in user code; their names are wired by TinyUSB at link time. Cite the TinyUSB docs at <https://docs.tinyusb.org/en/latest/reference/getting_started.html> and the class-specific pages.
- **Build** a CDC ACM virtual serial port. TinyUSB's CDC implementation expects two interfaces (one for the "Communication Class" with an interrupt-IN endpoint for the rarely-used notification stream, one for the "Data Class" with a bulk-IN/bulk-OUT pair for actual bytes). The class-specific functional descriptors inside the comm interface — Header (5 bytes), Call Management (5 bytes), Abstract Control Management (4 bytes), Union (5 bytes) — are defined by USB CDC 1.2 (free at <https://www.usb.org/document-library/class-definitions-communication-devices-12>) §5.2.3, pp. 36–48. The user-visible result is that `/dev/cu.usbmodem*` appears on macOS, `COM5` on Windows, `/dev/ttyACM0` on Linux. No driver install required since Windows 10 — the inbox `usbser.sys` covers it.
- **Build** a HID keyboard and mouse. HID is unique among USB classes in that the *report descriptor* — the bytes the host parses to understand what a "report" from your device means — is itself a small bytecode that you write. A standard 8-byte keyboard report is `(modifier byte, reserved, key1, key2, key3, key4, key5, key6)`; the matching report descriptor is ~63 bytes of HID-spec opcodes (Usage Page, Usage, Logical Min/Max, Report Size, Report Count, Input/Output). Cite HID 1.11 spec §6.2.2 (free at <https://www.usb.org/document-library/device-class-definition-hid-111>), pp. 23–33, and the HID Usage Tables 1.4 at <https://usb.org/document-library/hid-usage-tables-14>, pp. 53–63 for keyboard usage codes. The mouse report is 4 bytes (buttons, dx, dy, wheel) and its descriptor is ~50 bytes. We will copy both from the HID 1.11 spec's Appendix E examples and trace every byte.
- **Build** a Mass Storage Class device backed by a 64 KB RAM disk. MSC is the only class this week that runs a *protocol-inside-a-protocol*: USB-MSC's "Bulk-Only Transport" (USB BBB, see <https://www.usb.org/document-library/mass-storage-bulk-only-10>) wraps SCSI commands in 31-byte Command Block Wrappers and 13-byte Command Status Wrappers, transmitted over a bulk-IN/bulk-OUT pair. Your code answers four SCSI commands at minimum: INQUIRY (vendor/product strings), READ_CAPACITY_10 (block count and size), READ_10 (read N blocks starting at LBA), WRITE_10 (write N blocks). The RAM disk is a `uint8_t disk[131072]` array; we pre-format it as FAT12 with a `README.TXT` file already present. Cite Microsoft FAT specification "fatgen103.doc" (free at <https://academy.cba.mit.edu/classes/networking_communications/SD/FAT.pdf>) for the FAT12 layout.
- **Compose** the three classes into a single composite device using Interface Association Descriptors (IADs). IADs (USB 2.0 ECN, "Interface Association Descriptors", 2003-07-23, free at <https://www.usb.org/sites/default/files/iadclasscode_r10.pdf>) are 8-byte descriptors inserted before each *group* of interfaces that belong to one logical class. The composite device's configuration descriptor reads: IAD(CDC)+IF(CDC.comm)+CDC-funcs+EP(notify)+IF(CDC.data)+EP(bulk-in)+EP(bulk-out) + IAD(HID)+IF(HID)+HID-desc+EP(int-in)+EP(int-out) + IAD(MSC)+IF(MSC)+EP(bulk-in)+EP(bulk-out). 7 interfaces total in this build; the math is in Lecture 3.
- **Read** a USB packet capture and identify enumeration failures. Wireshark with USBPcap (Windows) or the built-in `usbmon` interface (Linux) or `PCAP_USB` (macOS via Apple's USB Prober) all show the raw control transfers during enumeration. A successful enumeration shows ~30 transactions on endpoint zero in the first 200 ms after attach. A failed one shows either a stall (`STALL` PID in response to a control SETUP) or a timeout (the host gives up after 5 retries per USB 2.0 §8.4.5, p. 213). We will deliberately break the descriptor in Exercise 3 and walk the resulting capture.
- **Measure** USB throughput. Full-speed USB is 12 Mbit/s on the wire, but per-transaction overhead (token + data + handshake) and the host's polling interval cap real-world throughput. A CDC ACM bulk-IN endpoint with `wMaxPacketSize=64` can sustain ~1.0 MB/s outbound on Linux, ~0.9 MB/s on Windows, ~1.1 MB/s on macOS (the variance is the host's `bMaxBurst` honoring). The mini-project benchmarks this by streaming a 1 MB pseudo-random buffer and timing the round trip.
- **Apply** the TinyUSB callback discipline. The two cardinal rules: (1) never block in a callback — TinyUSB calls them from its own task context and a 100 ms blocking call there hangs the entire device; (2) never call `tud_*_write` of more bytes than `tud_*_write_available()` returns — the underlying FIFO has fixed size and the excess is silently dropped. Both are enforced by code review in this course, not by any compiler diagnostic. Cite the TinyUSB API conventions doc at <https://docs.tinyusb.org/en/latest/reference/concurrency.html>.
- **Build** an end-to-end composite-device firmware: CDC echo at 115200-equivalent rate, HID keyboard that types a string on BOOTSEL press, MSC RAM disk with a FAT12 `README.TXT` and a `LOG.TXT` that the firmware appends to. Budget: enumerates cleanly on Linux, macOS, and Windows 10/11 without driver install; HID latency < 8 ms button-to-keystroke; MSC random-read throughput > 500 KB/s.

## Topics covered

- **USB as a tiered, polled bus.** USB is a tree topology with one host and many devices; the host is the only initiator (devices respond to host polls). USB 2.0 §4.1, p. 19. Full-speed (12 Mbit/s) frame interval is 1 ms; high-speed (480 Mbit/s) microframe interval is 125 µs. The RP2040's USB controller is full-speed only (RP2040 datasheet §4.1.1, p. 392). Every endpoint a device exposes has a polling rate the host respects; a bulk endpoint is polled "as fast as possible" with NAK throttling, an interrupt endpoint at `bInterval` ms (1–255), an isochronous endpoint exactly once per frame.
- **Endpoint zero and control transfers.** Every USB device has a bidirectional endpoint zero used exclusively for control transfers. A control transfer is three stages: SETUP (8-byte SETUP packet from host to device with bmRequestType, bRequest, wValue, wIndex, wLength), DATA (zero or more packets in the direction implied by bmRequestType, length up to wLength), STATUS (a zero-length packet in the opposite direction acknowledging completion). USB 2.0 §8.5.3, pp. 220–224. Standard, class-specific, and vendor-specific requests all flow through this pipe. The eight standard requests are listed in USB 2.0 Table 9-4, p. 251.
- **The four core descriptors.** Device, Configuration, Interface, Endpoint. Plus String (numeric-indexed UTF-16LE strings, §9.6.7, pp. 271–273) and class-specific (HID Report, CDC Functional, etc.) descriptors. The Device descriptor is fetched first and tells the host "I am one logical thing, vendor X product Y, with N configurations." The Configuration descriptor and its trailing descriptors together describe one *power and bandwidth profile* — most devices have only one configuration; multi-configuration devices are rare. The Interface descriptor groups endpoints belonging to one class; the Endpoint descriptor describes one direction of one pipe.
- **Descriptor parsing pitfalls.** The host parses the configuration's trailing descriptors with `wTotalLength` as the only length authority — if you set `wTotalLength=66` but the actual concatenated bytes are 75, the host either stops at 66 (losing the last 9 bytes' worth of interfaces/endpoints) or reads past the buffer (reading whatever is in your `.rodata` next). Every descriptor in the chain has `bLength` as its first byte, and the host advances by `bLength` per step; if any `bLength` is wrong, the host miscounts and the whole tree falls apart. We will demonstrate both failure modes in Exercise 3.
- **CDC ACM internals.** CDC defines several models; "Abstract Control Model" (CDC 1.2 §3.6.2) is the one we use because it gives us the bulk-IN / bulk-OUT pair we want for byte streams without any modem-control semantics getting in the way. The Communication interface's notification endpoint exists per spec but TinyUSB never actually sends on it in the ACM case; the host's driver is happy with the endpoint being declared and silent. CDC's `SET_LINE_CODING` request (baud, parity, stopbits, databits) reaches your code as `tud_cdc_line_coding_cb(itf, line_coding)` — you can ignore the values (USB doesn't actually run at 115200; the baud field is a host-side legacy bridge for tools that expect a serial port) or you can record them.
- **HID Report Descriptors.** A HID report descriptor is a sequence of "items", each 1–5 bytes, with a tag/type opcode and a payload. The opcode set is defined in HID 1.11 §6.2.2 (Item types: Main, Global, Local; Main: Input, Output, Feature, Collection, End Collection; Global: Usage Page, Logical Min/Max, Physical Min/Max, Unit, Report Size, Report Count, Report ID, Push, Pop; Local: Usage, Usage Min/Max, Designator). The report descriptor is parsed by the host into a *report format*; subsequent IN/OUT reports on the interrupt pipe are blobs of bytes that the host de-serializes against the format. HID Usage Tables 1.4 defines the *meaning* of the usage codes — Usage Page 0x01 is "Generic Desktop", Usage 0x06 inside that is "Keyboard", Usage Page 0x07 is "Keyboard/Keypad" with usages 0x04–0xE7 mapped to specific keys.
- **MSC bulk-only transport.** The host sends a 31-byte Command Block Wrapper (CBW: `dCBWSignature=0x43425355u, dCBWTag, dCBWDataTransferLength, bmCBWFlags, bCBWLUN, bCBWCBLength, CBWCB[16]`) on the bulk-OUT endpoint. The CBWCB is up to 16 bytes of SCSI command, the first byte of which is the SCSI opcode (0x12 INQUIRY, 0x25 READ_CAPACITY_10, 0x28 READ_10, 0x2A WRITE_10, 0x00 TEST_UNIT_READY, 0x1A MODE_SENSE_6, 0x1E PREVENT_ALLOW_MEDIUM_REMOVAL). The device responds with the data phase on bulk-IN or bulk-OUT, then a 13-byte Command Status Wrapper (CSW: `dCSWSignature=0x53425355u, dCSWTag (echo of CBW tag), dCSWDataResidue, bCSWStatus`). TinyUSB hides the CBW/CSW machinery; you implement the SCSI command callbacks (`tud_msc_inquiry_cb`, `tud_msc_capacity_cb`, `tud_msc_read10_cb`, `tud_msc_write10_cb`).
- **Composite-device descriptor math.** A CDC-only device has 2 interfaces. A HID-only device has 1 interface. An MSC-only device has 1 interface. The composite has 2 + 1 + 1 = 4 interface-slot positions but we add three IADs (1 per class group) so the configuration descriptor's `bNumInterfaces=4` and `wTotalLength = 9 (config) + 8 (IAD-CDC) + 9 (IF.comm) + 5+5+4+5 (funcs) + 7 (EP notify) + 9 (IF.data) + 7+7 (EP bulk) + 8 (IAD-HID) + 9 (IF.hid) + 9 (HID desc) + 7+7 (EP int) + 8 (IAD-MSC) + 9 (IF.msc) + 7+7 (EP bulk) = 161 bytes`. We compute this with `#define` arithmetic so the compiler verifies it.
- **The composite-device class triplet.** A composite device sets the *device-descriptor's* class triple to `(0xEF, 0x02, 0x01)` — "Miscellaneous, Common Class, Interface Association Descriptor" — to tell the host "look at my IADs to find the per-interface classes". Without this triple, Windows in particular fails to load drivers for the secondary interfaces. The triple is documented in the USB-IF "Interface Association Descriptor ECN" PDF (free).
- **TinyUSB's task model on the RP2040.** TinyUSB runs in a single function `tud_task()` that you call frequently from a thread or the main loop. It services the USB controller's interrupts, runs the enumeration state machine, calls your class callbacks, and returns. With FreeRTOS you usually create a dedicated task at high priority that runs `for (;;) tud_task(); vTaskDelay(1);` — the `vTaskDelay(1)` yields one tick so lower-priority tasks get CPU. Without FreeRTOS you call `tud_task()` inside your main loop next to your application logic. Cite <https://docs.tinyusb.org/en/latest/reference/concurrency.html>.
- **The RP2040's USB controller as a memory-mapped peripheral.** RP2040 datasheet §4.1, pp. 392–446. The controller has 16 endpoint pairs (32 endpoints total: 16 IN + 16 OUT), 4 KB of DPSRAM (dual-port SRAM accessible by both the CPU and the USB controller's transaction engine), and a register file that TinyUSB writes for you. The CPU's role at runtime is exclusively the descriptor work and the class-callback bodies — the line-level USB protocol (NRZI, bit-stuffing, CRC16, packet IDs) happens in silicon.

## A note on the on-device compile

The exercises and the mini-project are written to compile under the pico-sdk + FreeRTOS-Kernel build set up in Week 6, with TinyUSB enabled (pico-sdk includes TinyUSB as a vendored library at `lib/tinyusb/`; the build switch is `target_link_libraries(<target> tinyusb_device tinyusb_board)`). Each `.c` file has a comment block listing the pico-sdk libraries it expects to link and the GPIO pin map it assumes. The `mini-project/README.md` shows a worked CMakeLists for the full target; copy it into a fresh project directory, drop the `.c` and `.h` files in, ensure `tusb_config.h` enables the classes you need (`CFG_TUD_CDC=1`, `CFG_TUD_HID=1`, `CFG_TUD_MSC=1`), and `cmake -B build && cmake --build build` should produce a `.uf2`. On-device compile and flash is a student responsibility this week as in Week 6, 7, and 8.

The `tusb_config.h` is a header file TinyUSB reads at compile time to know which classes you have enabled and how many endpoints each gets. The mini-project ships an annotated `tusb_config.h` in its directory; copy it into your project root and adjust if you remove a class. A wrong `tusb_config.h` is the second-most-common cause of "device enumerates but no class endpoints work"; the first is a wrong `wTotalLength`.
